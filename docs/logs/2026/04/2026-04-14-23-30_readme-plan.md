# README Rewrite Plan

## Observations from best-in-class OSS READMEs

**Common patterns** (pydantic, uv, dspy):
1. **One-liner tagline** immediately under the project name
2. **Badges** — CI, coverage, PyPI version, Python versions, license
3. **Visual** — either a benchmark chart, architecture diagram, or logo
4. **"Why this exists" paragraph** — crisp, 2-3 sentences
5. **Quick Start** — copy-paste install + first run, under 10 lines
6. **Features list** — scannable, not walls of text
7. **Minimal code example** — show what a config looks like
8. **Links to deeper docs** — not the docs themselves
9. **Contributing / License footer**

**What the current arka README does wrong:**
- No badges
- No "why" / value proposition
- Quick Start is fine but bare
- "Current Implemented Path" is an internal developer dump, not user-facing
- "Example Configs" is a flat list with no context
- No architecture visual (we have a great mermaid diagram in docs/features.md but it's not in the README)
- No code/config example showing what the YAML looks like
- 39 lines but wastes them on low-signal content

## Proposed structure (target: ≤50 lines per AGENTS.md rule)

```
# arka (अर्क)

One-line tagline.

Badges row (CI placeholder, Python 3.12+, License MIT, coverage)

## Why arka

2-3 sentences: the problem (one-off scripts for synthetic data), the solution (declarative YAML pipeline), the result (reproducible, quality-filtered SFT datasets).

## Quick Start

3-step: install, set key, run example. Plus "what you get" one-liner.

## How it works

Compact inline pipeline flow (text, not mermaid — fits in 2 lines):
Seeds → Generate → Dedup → Filter → Output

Then 4-5 bullet features with one-line descriptions.

## Example config (minimal YAML snippet — 10-15 lines)

Show the simplest possible config.yaml that actually works.

## Docs & Examples

3-4 links: docs/features.md, docs/configuration.md, examples/README.md

## Development

just check, just test — 2 lines.

## License
```

This hits ~45-48 lines, stays under 50, and follows OSS best practices.

## Key decisions
- **No mermaid in README** — many renderers (PyPI, terminals) don't support it. Link to docs/features.md instead.
- **Show a real YAML snippet** — this is the product's interface; show it.
- **Badges are placeholders** — no CI pipeline yet, but the structure is there.
- **Drop the "Current Implemented Path" dump** — move to docs or STATUS.md.
- **Drop the flat example list** — link to examples/README.md instead.
