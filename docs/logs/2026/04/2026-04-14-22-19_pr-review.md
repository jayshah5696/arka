# PR review grouping and recommendation

I reviewed the open PR backlog and compared it to the current `main` branch.

## What I checked

Current `main` still lacks these changes:
- `src/arka/cli.py`: no `--dry-run`, minimal help text, no end-of-run summary.
- `src/arka/config/loader.py`: still returns raw Pydantic validation strings.
- `src/arka/pipeline/dedup_stages.py`: `NearDedupStage` is still the O(n^2) representative scan.
- `src/arka/pipeline/source_stages.py`: no seed size / length limits.
- `src/arka/pipeline/checkpoint.py`: no SQLite permission hardening.

I also checked the current API-key serialization behavior on `main` with `uv run python`: `model_dump(mode="json")` currently masks `SecretStr` as `"***"`; it does **not** emit plaintext keys.

## PR groups

### 1. Performance / NearDedup LSH duplicates
PRs: #7, #10, #12, #15, #18, #20, #25, #28

All are variants of the same idea: replace the O(n^2) near-dedup loop with LSH bucketing.

Recommendation:
- Keep **one** as the reference implementation.
- Best reference: **#20** (has tests and fixture updates).
- Alternative if you want the latest wording only: #28.
- Do **not** merge all of them; they overlap heavily on the same files.
- Prefer making one clean replacement PR from #20's logic and closing the rest.

### 2. DX / CLI / validation duplicates
Subgroups:
- Validation formatting: #5, #8, #23
- Run summary: #11, #22
- Dry-run / help text: #14, #16, #26

These overlap on `src/arka/cli.py`, `src/arka/config/loader.py`, and related tests.

Recommendation:
- Do **one consolidated DX PR** instead of merging these independently.
- Suggested sources:
  - validation formatting: **#8** or #23
  - run summary: **#22**
  - dry-run/help: **#16**
- Close the rest after consolidation.
- Avoid merging #14 as-is because it is a grab-bag overlap PR.

### 3. Security: API-key serialization duplicates
PRs: #9, #19, #21, #24, #27

These all address the same concern: excluding API keys from serialized config output.

Important current-state note:
- On current `main`, API keys are already stored as `SecretStr` and serialize to `"***"`, not plaintext.
- So the "CRITICAL plaintext leak" label appears stale relative to current HEAD.

Recommendation:
- Treat this as **cleanup / hardening**, not an emergency.
- If you want config dumps to omit `api_key` entirely rather than show `***`, keep **#21** as the best candidate because it also includes tests.
- Close #9, #19, #24, and #27 as duplicates.

### 4. Security: distinct hardening
PRs:
- **#13** SQLite permission hardening
- **#17** seed-size / prompt-injection / OOM hardening

Current `main` still lacks both.

Recommendation:
- **#17 is worth doing**; it covers a real missing safeguard and also adds `runs/` to `.gitignore`.
- **#13 is partially superseded by #17** because #17 also touches SQLite permissions.
- Review one design choice in #17: silent truncation of long seed text may be less desirable than explicit validation/failure.

### 5. Product features
PRs:
- **#1** Latent Density Sampling
- **#2** UMAP visualization command
- **#3** Privacy-aware guardrails

These are not duplicates of the above and need product decisions.

Recommendation:
- **#1**: keep for roadmap review; not obviously urgent.
- **#2**: do not merge blindly; it adds heavier analysis dependencies (`umap-learn`, `plotly`, `matplotlib`, `pandas`, `pyarrow`). Better as an optional analysis extra or separate tool.
- **#3**: do not merge as-is. It includes committed example run artifacts (`examples/runs/...`, `state.db`, output files). Salvage the core code/tests, but strip generated artifacts first.

### 6. Docs
PR: **#4**

Recommendation:
- Likely useful, but review README growth because project guidance says README should stay concise.
- Safer path: take `docs/configuration.md` and `docs/features.md`, keep README minimal.
