# AGENTS + rl-data-needed review

I read `AGENTS.md` and `docs/rl-data-needed.md` and checked the current Arka codebase.

## What the requirements doc is asking for

`docs/rl-data-needed.md` is a requirements doc for using Arka as the backbone for Humanize-RL.

It wants these main pieces:

1. `AIifyStage` — turn human text into intentionally AI-sounding text
2. `RubricScoringStage` — score text on a 10-dimension humanness rubric
3. `HumanizeStage` — rewrite AI-ish text into more human text
4. `PairFilterStage` — keep only good before/after pairs
5. `IFDStage` — instruction-following quality gate
6. `RecordScores` extensions — humanness fields

## What already exists in Arka

The repo already has useful building blocks:

- `src/arka/records/models.py`
  - `RecordScores`
  - lineage + stage events
- `src/arka/labeling/engine.py`
  - batch judge execution
- `src/arka/labeling/rubric.py`
  - rubric loading/validation
- `src/arka/pipeline/filter_stages.py`
  - `LabelingQualityFilterStage`
- `src/arka/pipeline/ifd_stage.py`
  - existing IFD filtering implementation
- `src/arka/pipeline/generator_stages.py`
  - pattern for LLM-backed stage execution and checkpointing

## Biggest gap

The current pipeline wiring is still a **fixed source/generator/dedup/filter flow** via `src/arka/pipeline/stage_builder.py`.

But `docs/rl-data-needed.md` needs a **repeatable stage sequence** like:

- rubric scoring
- aiify
- rubric scoring again
- humanize
- rubric scoring again
- pair filter
- ifd

So the first practical unblocker is not just “add one stage” — it is enabling a configurable stage list or equivalent stage chaining model.

## Important mismatch found

The doc says IFD is a stub, but current Arka already has IFD support in `src/arka/pipeline/ifd_stage.py`.

However, the formula in the doc is:

`exp(mean_conditional_lp - mean_unconditional_lp)`

while current code computes:

`conditioned.mean_logprob - unconditioned.mean_logprob`

So IFD is **partially implemented but not aligned with the new spec**.

## Recommended next steps

### Step 0 — pipeline plumbing
Add support for ordered, repeated stage configs.

Files likely touched:
- `src/arka/config/models.py`
- `src/arka/config/loader.py`
- `src/arka/pipeline/stage_builder.py`

Goal:
- support a `stages:` list with typed stage configs
- preserve current config format for backward compatibility if possible

### Step 1 — extend record scoring model
Update `RecordScores` so humanness data has a typed home.

File:
- `src/arka/records/models.py`

Add fields like:
- `humanness`
- `humanness_per_dim`
- `humanness_checklist`
- `humanness_reasoning`

This is small and low-risk, and it unblocks later stages.

### Step 2 — implement `RubricScoringStage` first
This should be the first new stage because everything else depends on it.

Reuse:
- `RubricLoader`
- `LabelingEngine`
- current labeling/filter stage patterns

Likely need:
- a stage that scores without necessarily dropping
- optional min/max threshold filtering
- ability to tag/store multiple rubric passes in lineage/stage artifacts

### Step 3 — implement `AIifyStage`
Use the existing LLM-backed generator stage patterns as the template.

Need to decide whether output is:
- a transformed `ConversationRecord`, or
- a richer/new record type that preserves source + aiified text together

My recommendation: use a richer/new record model early, because pair comparison will be awkward otherwise.

### Step 4 — implement `HumanizeStage`
Very similar mechanics to `AIifyStage`, but prompt is driven by rubric dimensions.

Dependency:
- rubric scoring output must be available in a structured way

### Step 5 — implement pair representation + `PairFilterStage`
Before filtering pairs, decide how Arka represents them.

You likely need either:
- a dedicated pair/preference-style record, or
- stable metadata links between related records

This is required for:
- delta scoring
- semantic similarity
- length ratio checks
- dimension improvement counts

### Step 6 — reconcile IFD with the new RL spec
Update existing IFD code to match the desired formula and thresholds.

Files:
- `src/arka/pipeline/ifd_stage.py`
- tests in `tests/unit/test_ifd_scorer.py`

## Suggested implementation order in this repo

Practical order for Arka specifically:

1. stage-list config/builder support
2. `RecordScores` extensions
3. `RubricScoringStage`
4. `AIifyStage`
5. `HumanizeStage`
6. pair record/model + `PairFilterStage`
7. IFD alignment
8. integration fixture for the humanize pipeline

## Test plan

Follow the repo’s TDD rule for logic changes.

Add first:
- config loader tests for `stages:`
- stage builder tests for repeated stages
- unit tests for `RubricScoringStage` with fake judge client
- unit tests for `AIifyStage` / `HumanizeStage` with fake LLM outputs
- roundtrip tests for new `RecordScores` fields through parquet/jsonl
- updated IFD tests for new formula

## Bottom line

The next real move is:

**implement pipeline stage-list plumbing + `RecordScores` extensions, then build `RubricScoringStage` first.**

That gives the foundation for `AIifyStage`, `HumanizeStage`, and pair filtering without painting the pipeline into a corner.
