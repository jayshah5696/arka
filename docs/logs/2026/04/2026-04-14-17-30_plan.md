# Re-read of `docs/rl-data-needed.md`, revised plan, and TransformStage implementation

I re-read the updated `docs/rl-data-needed.md`, checked it against the current Arka codebase, and then implemented the first TDD slice.

## Updated architectural reading

The requirements doc is now clearly saying:

- **Do not put humanize-rl-specific logic into Arka**
- Add **generic framework primitives** that many pipelines can use
- Humanize-RL should mostly be:
  - prompt templates
  - rubric YAML
  - config
  - small project-specific glue only where necessary

That changes the plan.

## What I concluded before coding

Arka already has a workable spine:

- source stages
- normalize
- generator stages
- dedup
- filters
- labeling engine
- checkpointing/reporting

The best next move is **not** to build humanize-only stages and **not** to jump straight into a full DAG/workflow engine.

Instead, Arka needs better **composable primitives**:

1. **Transform primitive**
2. **Score primitive**
3. **Selection/filter primitive**
4. **Pair/delta primitive**
5. **Client capability support for reward/logprob scoring**

## Hard constraints from current code

### 1. Pipeline shape is still linear and fixed
`src/arka/pipeline/stage_builder.py` still builds:
- source
- generator
- dedup
- filters

So a full one-run flow like:
- transform → score → transform → score → pair filter

is not natural yet.

But the updated requirements explicitly say:
- two runs + glue is acceptable now
- one-run multi-step is a stretch goal

So we do **not** need to solve a full stage DSL first.

### 2. Scoring is filter-coupled
Current score-related stages:
- `LabelingQualityFilterStage`
- `IFDFilterStage`

Both mix:
- compute score
- apply threshold
- drop records

Longer-term, Arka should move toward:
- **score stages annotate records**
- **filter/select stages consume those scores**

### 3. Current labeling assumes `(instruction, response)`
`LabelingEngine` and prompt-building currently expect an instruction/response pair.

But the updated doc points out a need to score:
- standalone text
- arbitrary record fields
- transformed text only

So Arka needs a more general scoring target abstraction later.

### 4. IFD is more incomplete than it first appears
`src/arka/pipeline/ifd_stage.py` exists, but the live client path is not implemented:

- `provider_supports_sequence_scoring()` returns `False`
- `LLMClient.score_response()` raises `unsupported_capability`

So real IFD needs client work, not just stage math cleanup.

### 5. Stage-local model overrides will matter
The requirements doc wants different models/endpoints for:
- transform
- rubric judge
- reward model
- IFD/logprobs

Current config is still centered on one top-level `llm`.

That remains a real framework gap for the next slice.

## Revised plan

## Phase 1 — Highest-leverage generic primitives

### 1. Add `TransformStage` / transform generator first
This is the most important generic capability.

Why first:
- fits the updated philosophy
- unlocks rewrite/paraphrase/style-transfer/translation workflows
- fits current architecture with less disruption than a full stage DSL

### 2. Split “score” from “filter”
Before adding reward model or pair delta logic, Arka should evolve toward:
- `LabelingScoreStage`
- `RewardModelScoreStage`
- `IFDScoreStage`

with thresholding/filtering separate.

### 3. Generalize scoring targets
Add support for scoring:
- conversation pair
- standalone text field
- configurable field pairs

### 4. Add stage-local model overrides / model refs
Needed for:
- cheap transform model
- expensive rubric judge
- reward model endpoint
- logprob-capable IFD model

## Phase 2 — Scoring capability completion

### 5. Implement real IFD support
After score abstraction exists:
- implement live logprob/sequence scoring in `LLMClient`
- reconcile capability semantics
- update formula/tests

### 6. Add `RewardModelScoringStage`
After scoring abstraction + model override support.

## Phase 3 — Comparison and selection

### 7. Add `PairDeltaFilterStage`
After:
- transform exists
- scores are written separately from filtering
- preserved originals / lineage are reliable

### 8. Add `CompositeSelectStage`
Once Arka has:
- `quality`
- `reward_model`
- `ifd`

## Phase 4 — Orchestration and heuristics

### 9. Add multi-step `stages:` sequencing
Important, but after the primitive stages exist.

### 10. Add extra cheap heuristics
Possible later additions:
- sentence variance
- repetition within a document
- formatting heuristics

## Current behavior of the new transform stage

Supported field paths right now:
- `payload.instruction`
- `payload.response`
- `payload.system`

This is intentionally narrow and typed for the first slice.

When `preserve_original: true`, the original target-field text is stored as:

```json
{"transform_original":{"field":"payload.response","text":"Old response"}}
```

inside `payload.system`.

That is not the final abstraction forever, but it is a small, typed, reversible first step.

## What this unlocks

Arka can now express generic rewrite-style pipelines without custom humanize-specific stages.

Examples this first slice can support with config:
- rewrite response tone
- paraphrase instruction
- style transfer of response text
- simple AIify/Humanize prototypes using prompt templates

## Best next implementation step

The next slice should be:

1. **stage-local model overrides / model refs**
2. **separate score from filter** by introducing a scoring stage
3. **generalize LabelingEngine inputs** beyond `(instruction, response)`

If doing only one immediate follow-up, the best one is:

> **introduce a `LabelingScoreStage` and begin separating annotation from threshold filtering**

That keeps the framework moving in the generic direction established by the updated requirements.
