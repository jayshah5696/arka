## 2024-05-18 - LSH Bucketing in NearDedupStage / Learning: The O(n^2) bottleneck in `NearDedupStage` was caused by comparing each incoming record's MinHash signature against every previously seen cluster representative in the worst case (when the loop doesn't break early). / Action: Implemented Locality Sensitive Hashing (LSH) bucketing. Added `LSHIndex` utilizing `num_bands` and `rows_per_band` (configurable, default 16x8 for 128 hashes). Now we only query the LSH bands to retrieve candidate representatives, reducing the time complexity to O(n) average case. This requires updating all config schemas across YAML files to accommodate `extra="forbid"`.
